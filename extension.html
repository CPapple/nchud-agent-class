Skill Design 6 分

Skill 不是收藏 Prompt，
而是把你的工作方法寫成 SOP

好的 Skill 能讓不同的人、不同時間、不同資料，都走出相近品質的結果。

何時啟動使用情境
也寫不適用情境
＋
需要什麼檔案、主題
日期、對象
＋
怎麼做步驟順序
工具與查核
＋
交付格式欄位、篇幅
語氣、檔型
＋
遇錯怎辦缺資料先問
高風險交給人
判斷標準：新同事只看這份 Skill，能不能照著完成工作？

依據：Agent Skills 將 Skill 定義為可攜式的指令、腳本與資源組合，並以漸進揭露方式按需載入。Anthropic Engineering

Trigger Quality 6 分

Skill 做得很好，為什麼 Agent 沒想到要用？

description 像門口招牌：太模糊，找不到你；寫太廣，什麼事都闖進來。

太模糊
「協助處理館務資料。」

問題：什麼資料？何時該用？輸出什麼？Agent 無法判斷。

→
可觸發、可排除
「當使用者要把新書清單整理成每週館藏通報時使用；輸出書名、作者、三句摘要與推薦對象。單本書目查詢不要使用。」
01

寫使用者意圖

不要只寫內部技術；寫「他想完成什麼」。

02

正例與反例

準備 5 個該觸發、5 個不該觸發的說法。

03

用真實口語測

學員不一定會講 Skill 名稱，仍要找得到。

不是把 Skill 名稱背起來才叫會用。真正好用的 Skill，是同事用平常說話的方式交辦，它也知道該拿哪一本 SOP。

依據：Agent Skills 官方建議以 should-trigger／should-not-trigger 查詢測試 description。Optimizing skill descriptions

Prompt Iteration 8 分

不要追求神 Prompt；
用三次修改把不可控變可驗收

V1 · 只有願望

「幫我整理今天的 AI 新聞。」

不知道來源、讀者、則數、時間範圍，也沒有輸出標準。

V2 · 補齊工作條件

「找過去 24 小時 5 則 AI 新聞，給非技術館員閱讀。」

範圍清楚了，但仍可能選到重複或沒有來源的內容。

V3 · 加上驗收規格

「每則含標題、60 字摘要、原始連結、館務影響；去重，無可靠來源則省略。」

完成條件可看、可查、可改，才適合存成 Skill 或交給 Cron。

優化順序：先看失敗在哪裡 → 只補那一條 → 用同一批案例重跑。
Context Engineering 7 分

近期的重點已從「怎麼問」
移到「讓它在工作時看見什麼」

Prompt 只是其中一格。Agent 的結果還受資料、工具、記憶、範例與成功標準共同影響。

任務 Task這次到底要完成什麼；不要把所有背景都塞成命令。
資料 Data規章版本、讀者對象、日期範圍、可信來源；只拿此刻需要的。
工具 Tools允許搜尋、讀檔或寄送嗎？工具越多不一定越好。
記憶 Memory長期偏好放 Memory；一次性需求留在這次任務。
驗收 Check要求有來源、固定欄位、不得猜測；說清楚何時要停下來問人。
以前我們像在研究一句咒語；現在更像替新同事準備桌面：正確文件放桌上、只給需要的工具、把交付規格貼在旁邊。

依據：Context engineering 是針對 system instructions、tools、外部資料、歷史訊息等整體脈絡的配置；核心是高訊號、少雜訊。Anthropic Engineering

Evaluation 6 分

Prompt 優化最常漏掉的一步：
先寫「怎樣算做好」

不要只憑「這次看起來不錯」。準備一小包固定測試題，每次改 Skill 後都重跑。

1

正常案例

資料完整時，格式、語氣與欄位都正確。

2

缺資料案例

沒有日期或檔案時，會先詢問而不是猜。

3

邊界案例

個資、付費、發布或刪除動作會停下來。

4

反例

不符合任務時，不應亂啟動這個 Skill。

固定 8–10 題用真實工作語句
→
每次改一件事描述、範例或規格
→
比較結果有進步，也沒有害到舊案例
AI 的回答有變動性，所以不要只問「這一次漂亮嗎」，而要問「換五種真實資料，它還守不守規矩」。

依據：Agent eval 應定義 task、trial、grader 與 outcome，並以多次試跑觀察穩定性。Demystifying evals for AI agents

Cron 不是「幾點叫醒 AI」，
而是一份無人值守的交辦單

何時固定時間
或一次性提醒
＋

搭配既有 Skill
＋

或通訊平台

不自行擴權

適合排程

重複頻率高、輸入來源穩定、輸出容易查核、失敗可逆、收件人清楚。

先不要排程

會付款、會公開發布、會刪資料、涉及敏感個資、錯一次就難以回復。

排程前先檢查：如果今晚三點它做錯，而明早才被發現，最壞會怎樣？答案不能接受，就不要全自動。

Hermes 官方目前支援一次性／週期任務、Skill-backed jobs、多個 Skills、指定工作目錄與多種傳送出口。Hermes Scheduled Tasks

Cron Recipes 8–12 分

最值得先做的六種通用排程

優先選「蒐集、整理、提醒、比對、草擬」；把「核准、發布、刪除」留給人。

每日主題情報

可自動

固定來源抓新訊息，去重後產出摘要與連結。

截止日雷達

可自動

每天檢查活動、採購或計畫期限，只推送未完成項。

每週進度簡報

送前確認

彙整本週完成、進行中、風險與下週計畫。

內容變動巡檢

可自動

比對網頁或文件版本，只列出新增、刪除與異動。

待辦事項追蹤

人工補狀態

定期彙整未完成事項，標示負責人與到期日。

對外內容草稿

不要直發

可以定時產生草稿；正式發布前保留人工核准。

Safe Autonomy 6 分

不要只分「自動／不自動」；
替每件任務選一個安全刻度

0 · 建議AI 只提供想法
人決定也人執行
1 · 草擬AI 產出初稿
人檢查後採用
2 · 執行後通知低風險、可回復
留下紀錄供抽查
3 · 需核准才執行付費、公開、刪除
敏感資料與權限變更

重點不是一直跳確認框

確認太頻繁會形成疲勞；應直接限制它能接觸的資料、工具、收件人與可執行動作。

先限制最壞後果

測試資料夾、唯讀權限、固定白名單、草稿匣、可回復紀錄，比一句「請小心」可靠。

我們不是靠 AI 自律，而是像管理新進同仁：先給剛好夠用的鑰匙，重要動作仍要簽核。

近期安全討論將重點放在限制 blast radius：透過 sandbox、VM、網路出口與權限邊界，限制 Agent 能造成的最壞影響。How we contain Claude across products；Hermes Security Policy

Agent Profiles 7 分

Hermes One 的 Agents，
實際上是不同 Hermes Profiles

每個 Profile 都是一個獨立工作角色，不只換名字；它擁有自己的身分、記憶、技能與自動化設定。

IDENTITY

SOUL.md

角色定位、思考原則、語氣與溝通方式。

KNOWLEDGE

Memory＋Skills

各自累積的偏好、經驗與專用工作方法。

SETUP

Model＋Tools

每個角色可使用不同模型、工具與設定。

WORK

Cron＋Sessions

各自的排程、對話、日誌與 Gateway 狀態。

教學上可把它理解成「同一套 Hermes 裡的不同專業角色」：研究員、查核員、編輯、專案助理，各自有自己的工作設定。

官方定義：Profile 有獨立的 config.yaml、SOUL.md、sessions、memory、skills、cron 與 gateway state；但 Profile 本身不是安全沙箱。Hermes Profiles · Hermes Desktop

Discover → Agents 8 分

探索 Agents：
直接建立別人設計好的專業角色

Discover 的 Agent 不是換模型，也不是安裝一個 Skill；它會建立新的 Profile，並套用該角色完整的工作人格與原則。

1 · 探索Discover → Agents

→

例如 API Designer
→
3 · 成為新 Agent複製預設 Profile
套用新的角色人格
REGISTRY EXAMPLE

API Designer

把 API 視為「契約，也是開發者的使用者介面」；先研究現有規則，再設計 REST、RPC 或 GraphQL。

一致性優先為呼叫者設計避免破壞相容性
它不只是一段角色名稱

工作原則＋方法＋交付規格

要求具體定義 endpoints、schemas、errors、pagination、auth、idempotency 與 versioning，最後交付 OpenAPI／JSON Schema 式規格與成功／錯誤範例。

背後機制：Hermes One 會複製預設 Profile，再把 Registry Agent 的 AGENT.md 寫成新 Profile 的 SOUL.md。下載 Agent 的核心，就是引入一套設計好的 Soul。

Hermes One 原始碼將 Agents 列為 Discover registry 類型，安裝動作是 Create；專案文件明載「建立由 default clone 的新 Profile，並將 Agent 的 AGENT.md 寫成 SOUL.md」。Discover 原始碼 · Registry 安裝模型 · Hermes Registry

What people discuss now 8 分

現在業界在談的，已不只是「模型更強」

1

Context 是否太肥

不是塞越多越聰明；高訊號、按需載入、定期壓縮才穩。

2

Skill 能否被正確叫到

description、正反例與漸進揭露，成為 Skill 設計核心。

3

Skill 生態與信任來源

Skill 變得可分享、可安裝，也代表安裝前必須審查來源與腳本。

4

Agent 到底有沒有變好

從「Demo 看起來很神」走向固定案例、trace 與 outcome 的評估。

5

背景任務的可觀測性

排程要能暫停、重跑、看歷史、知道失敗，不能設定後就忘記。

6

Autonomy 的爆炸半徑

與其一直問「你確定嗎」，更重要的是限制資料、工具與可造成的後果。

2026 年談 Agent 的成熟度，不只看它能不能跑，而是能否回答：它為什麼啟動、看了什麼、做過什麼、失敗後怎麼停。

近期 Hermes 發展也反映相同方向：縮減預設 Skill 以降低雜訊、Skill 使用追蹤、Cron 與 Skill 串接、排程內容安全掃描。Hermes Agent Releases
